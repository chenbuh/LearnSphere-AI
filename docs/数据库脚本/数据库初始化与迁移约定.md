# 数据库初始化与迁移约定

本文档用于统一根目录 SQL 脚本与 `docs/数据库脚本` 下历史更新脚本的使用方式，降低本地初始化、测试补表和 Linux 宝塔上线时的执行歧义。

## 一、当前 SQL 文件分布

### 1. 根目录

当前根目录存在以下数据库相关脚本：

- `check_tables.sql`
  - 用途：快速检查部分核心表结构是否存在。
  - 当前内容较轻量，更像排障辅助脚本，不应视为完整初始化脚本。
- `migration_ai_governance.sql`
  - 用途：初始化 AI 治理相关表，例如 `system_prompt`、`ai_generation_log`。
- `migration_ai_governance_v2.sql`
  - 用途：AI 治理的后续结构扩展，建议在 `migration_ai_governance.sql` 之后执行。
- `migration_ai_feedback_pool.sql`
  - 用途：AI 反馈池或审核池相关结构。
- `migration_add_user_id.sql`
  - 用途：为既有表补 `user_id` 等关联字段。
- `migration_add_token_stats.sql`
  - 用途：补 AI Token / 配额统计相关字段或表结构。
- `migration_add_quota_configs.sql`
  - 用途：补 AI 配额配置相关结构。
- `seed_prompts.sql`
  - 用途：插入系统提示词或初始化 AI 提示词数据。

### 2. `docs/数据库脚本`

当前目录下存在：

- `update_2.2.0_add_points.sql`
- `update_2.3.0_achievement.sql`

这类脚本更适合视为“历史版本升级脚本”，用于已经存在业务库时的增量升级，不建议当成全量初始化入口。

## 二、推荐执行顺序

### 场景 A：全新本地库初始化

适用于本地开发、测试环境首次建库。

建议顺序：

1. 先通过后端基础建表能力或已有初始化脚本完成主业务表落库。
2. 再执行 AI 治理与扩展脚本：
   - `migration_ai_governance.sql`
   - `migration_ai_governance_v2.sql`
   - `migration_ai_feedback_pool.sql`
3. 再执行统计与配置增强脚本：
   - `migration_add_user_id.sql`
   - `migration_add_token_stats.sql`
   - `migration_add_quota_configs.sql`
4. 最后执行初始化数据：
   - `seed_prompts.sql`
5. 如需快速抽查结构，可执行：
   - `check_tables.sql`

### 场景 B：已有环境补迁移

适用于测试库、预发布库、线上库已有历史数据时的补结构操作。

建议原则：

1. 先确认当前数据库版本和表结构差异。
2. 仅执行缺失的迁移脚本，不要重复执行整套脚本。
3. 先执行结构变更，再执行种子数据脚本。
4. 变更前先备份数据库或至少备份被修改表。

### 场景 C：历史版本升级

适用于已经按旧版本运行的环境。

建议顺序：

1. 先执行根目录下与当前变更直接相关的迁移脚本。
2. 再按版本顺序执行 `docs/数据库脚本` 中的历史升级脚本。
3. 执行完后，再跑一次关键业务验收和后台配置检查。

## 三、执行约定

### 1. 新增脚本命名规则

后续新增 SQL 脚本建议使用下面两类命名：

- 全局增量迁移：`migration_YYYYMMDD_变更说明.sql`
- 版本升级脚本：`update_x.y.z_变更说明.sql`

不要继续使用语义过宽的名字，例如只叫 `update.sql`、`new.sql`、`temp.sql`。

### 2. 单个脚本职责尽量单一

建议一个脚本只做一类事情：

- 结构变更
- 初始化数据
- 回填数据

如果必须混合，至少在文件头注释里写清楚执行前提、影响表和是否可重复执行。

### 3. 尽量保证可重复执行

推荐做法：

- `CREATE TABLE IF NOT EXISTS`
- `CREATE INDEX` 前先确认是否已存在
- 初始化数据优先使用 `INSERT IGNORE` 或幂等写法

这样可以降低测试环境重复执行的风险。

## 四、上线前检查清单

执行数据库脚本前，至少确认以下内容：

- 当前数据库实例、库名、账号是否正确。
- 已备份目标库或目标表。
- 已确认脚本执行顺序。
- 已确认脚本是否包含不可逆数据删除。
- 已确认本次部署对应的后端代码版本已经同步。

## 五、建议补充项

当前仓库已经有一批可用脚本，但还缺两类更正式的资料：

- 全量初始化脚本
  - 用于新环境一键建库，而不是依赖多个迁移脚本拼接。
- 数据库版本记录表
  - 用于记录某个环境已经执行过哪些迁移，避免重复执行或漏执行。
