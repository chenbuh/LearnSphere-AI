## 一、扫描结论

整体来看，这个项目已经具备比较好的产品雏形，技术栈也不落后：前端采用 `Vue 3 + Vite`，后端采用 `Spring Boot 3`，并且已经接入了 `PWA`、`Actuator`、`Prometheus`、`Resilience4j`、限流与权限控制，说明项目不是“从零搭的 demo”，而是已经进入了“要开始做工程化收口”的阶段。

这次结合仓库现状看，后续最值得投入的优化方向，不是继续堆新功能，而是优先做下面四类事情：

1. **瘦身资源与收敛构建产物**：前端本地题库和静态资源偏重，发布物也直接进入后端源码目录，长期会拖慢构建、提交和部署。
2. **拆分超大模块**：后端 AI 生成服务、前台学习页、后台治理页都出现了“大文件承载太多职责”的迹象。
3. **补齐质量门禁**：测试覆盖、Lint、格式化和 CI 校验还不够完整，容易在后期迭代中引入回归问题。
4. **标准化部署与数据变更**：数据库初始化仍偏“启动即修库”的方式，适合快速开发，但不适合多人协作和生产演进。

## 二、当前最明显的问题点

- `frontend-vue/src/data` 下共有 **47 个数据文件**，总体积约 **14.41 MB**，本地题库资源偏重。
- `frontend-vue/src/data/tem8_words.js` 与 `frontend-vue/src/data/tem8_words_updated.js` 内容完全一致，存在可清理的重复数据。
- `backend/src/main/resources/static` 下已有前台/后台构建产物，共 **291 个静态文件**，总体积约 **7.59 MB**，说明构建结果已经直接进入后端源码目录。
- `backend/src/main/java/com/learnsphere/service/impl/AIGenerationServiceImpl.java` 约 **2109 行**，AI 业务逻辑已明显过重。
- `backend/src/main/java/com/learnsphere/config/DatabaseInitializer.java` 约 **639 行**，承担了大量“建表 + 补字段 + 修数据”的职责。
- 前台页面 `frontend-vue/src/views/GrammarView.vue` 约 **1713 行**、`frontend-vue/src/views/ListeningView.vue` 约 **1690 行**、`frontend-vue/src/views/VocabularyView.vue` 约 **1454 行**，页面逻辑偏集中。
- 后台页面 `admin-vue/src/views/AIGovernance.vue` 约 **2031 行**，后续维护成本会越来越高。
- 测试数量不均衡：后端业务测试 **4 个**，前台业务测试 **2 个**，后台业务测试 **0 个**。
- `.github/workflows/ci.yml` 已存在，但目前前端/后台主要执行构建，质量门禁还不够完整。
- 仓库中暂未发现统一的 `ESLint / Prettier / Checkstyle / Spotless` 配置，代码规范更多依赖人工自觉。
- `frontend-vue` 与 `admin-vue` 的 `vue`、`vite`、`axios` 版本并不完全一致，后续维护会有额外心智成本。
- 仓库里暂未看到正式的 `Dockerfile` / `docker-compose`，部署标准化仍有提升空间。

## 三、建议先创建的优化任务

下面按优先级整理为可以直接推进的任务清单。

### P0：建议优先处理

- [ ] **任务 1：收敛前端题库与静态资源体积**
  - 目标：降低首包压力、减少构建时间、减轻仓库体积增长。
  - 建议动作：
    - 将 `frontend-vue/src/data` 的题库数据按考试类型、难度、模块拆分为可按需加载的资源。
    - 重复数据先清理，至少先处理 `tem8_words.js` 与 `tem8_words_updated.js` 这类重复文件。
    - 中长期建议把大题库迁移到后端接口、对象存储或 CDN，而不是长期放在前端源码中。
  - 预期收益：用户首次打开更快，前端构建更轻，内容迭代也更方便。

- [ ] **任务 2：停止将构建产物长期保留在源码目录中**
  - 目标：减少脏提交、避免旧资源残留、让发布流程更清晰。
  - 建议动作：
    - 明确 `frontend-vue`、`admin-vue` 的构建产物由 CI/CD 生成，再复制到后端发布目录。
    - 清理 `backend/src/main/resources/static` 内历史构建残留，避免多次 build 后旧 hash 文件持续堆积。
    - 将“源码”和“构建输出”职责彻底分开。
  - 预期收益：Git 历史更干净，部署包更可控，线上回滚也更清楚。

- [ ] **任务 3：拆分 AI 核心服务，降低单类复杂度**
  - 目标：提升 AI 相关代码可读性、可测试性和可扩展性。
  - 建议动作：
    - 将 `AIGenerationServiceImpl.java` 拆为至少几类职责：提示词组装、模型路由、配额扣减、降级兜底、日志记录、指标埋点。
    - Controller 保持“薄”，复杂编排尽量沉到专门的 orchestrator/service 中。
    - 与 AI 相关的 DTO、策略、模板、限流和审计规则，尽量分包管理。
  - 预期收益：后面新增模型、切换供应商、增加配额策略时，改动范围更小。

- [ ] **任务 4：把数据库变更改为正式迁移机制**
  - 目标：让数据库演进可追踪、可回滚、可多环境复现。
  - 建议动作：
    - 逐步用 `Flyway` 或 `Liquibase` 取代 `DatabaseInitializer.java` 中的大量启动时 SQL。
    - 将根目录零散的 SQL 脚本统一收口到后端 migration 目录管理。
    - 把“初始化建库”和“版本迁移”区分开，不再依赖应用启动时自动修表。
  - 预期收益：避免线上环境启动即改库带来的不确定性，也更适合团队协作。

### P1：建议在 P0 后紧接着做

- [ ] **任务 5：补齐测试、Lint 与格式化门禁**
  - 目标：把“能跑”提升为“可持续迭代”。
  - 建议动作：
    - 前台和后台都补 `ESLint + Prettier`。
    - 后端补 `Spotless` 或 `Checkstyle`，至少形成基础统一规范。
    - CI 中除了构建，还应执行前端测试、后台测试、后端测试与必要的静态检查。
    - 优先补高风险模块测试：登录鉴权、AI 生成、学习记录、管理后台治理配置。
  - 预期收益：后续改动更有安全感，减少“修一个地方坏一片”的情况。

- [ ] **任务 6：拆分前台大页面与后台治理页**
  - 目标：降低页面耦合度，提升功能扩展效率。
  - 建议动作：
    - 将 `GrammarView.vue`、`ListeningView.vue`、`VocabularyView.vue` 拆分为“页面容器 + 状态 composable + 展示组件”。
    - 将 `AIGovernance.vue` 拆为筛选区、表格区、详情抽屉、操作弹窗、统计卡片等独立模块。
    - 页面状态尽量从组件本身迁移到 composable 或 store，模板中少放业务判断。
  - 预期收益：单文件不再过大，联调和排障效率会显著提高。

- [ ] **任务 7：统一前端工程管理方式**
  - 目标：减少版本漂移和重复维护成本。
  - 建议动作：
    - 把根目录、`frontend-vue`、`admin-vue` 的 Node 侧依赖统一规划，考虑使用 `npm workspaces` 或 `pnpm workspace`。
    - 收敛 `vue`、`vite`、`axios` 等公共依赖版本。
    - 抽公共请求封装、通用组件、鉴权逻辑和常量定义，避免双端重复维护。
  - 预期收益：升级依赖时更顺手，也更适合后面继续扩展客户端。

- [ ] **任务 8：标准化部署与环境管理**
  - 目标：降低部署门槛，减少环境差异问题。
  - 建议动作：
    - 补齐正式的 `Dockerfile` / `docker-compose` 或等效部署方案。
    - 前台、后台分别补 `.env.example`，避免新成员直接复制真实配置文件。
    - 区分开发、测试、生产环境的配置来源和注入方式。
  - 预期收益：新环境落地更快，迁移服务器或交接项目更省心。

### P2：适合在系统稳定后持续推进

- [ ] **任务 9：完善监控、告警与 AI 成本看板**
  - 目标：把已有监控基础真正用起来。
  - 建议动作：
    - 基于现有 `Actuator + Prometheus` 补齐 Grafana 看板。
    - 对 AI 请求耗时、失败率、配额消耗、降级触发次数、热点接口设置阈值告警。
    - 前端增加 Web Vitals 或关键路径性能采样。
  - 预期收益：问题能更早暴露，AI 成本也更容易被量化管理。

- [ ] **任务 10：补强项目文档与架构说明**
  - 目标：提升项目可接手性与协作效率。
  - 建议动作：
    - 把“系统模块图、部署图、数据库迁移策略、前后端接口约定”整理成固定文档。
    - 把“哪些资源由前端内置、哪些资源由后端提供、哪些资源可通过管理后台维护”写清楚。
    - 为关键模块增加维护说明，例如 AI 治理、学习记录、权限控制、题库导入。
  - 预期收益：后续无论是自己回头看，还是交给别人接手，成本都会低很多。

## 四、推荐执行顺序

如果按“投入产出比”来排，我建议按下面顺序推进：

1. **先做资源瘦身和构建产物治理**
2. **再拆 AI 服务与大页面**
3. **然后补测试、Lint、CI 门禁**
4. **最后推进数据库迁移标准化与部署标准化**

这样安排的好处是：

- 第一阶段先解决最直观的性能和仓库膨胀问题；
- 第二阶段解决未来最容易失控的复杂度问题；
- 第三阶段把质量门槛立起来；
- 第四阶段再把运维和数据演进做成长期稳定机制。

## 五、一句话建议

这个项目现在最需要的不是“再加更多功能”，而是把已经做出来的功能**工程化、模块化、标准化**。只要把资源管理、AI 核心服务、数据库迁移、测试门禁这四块补起来，项目的可维护性会明显上一个台阶。
