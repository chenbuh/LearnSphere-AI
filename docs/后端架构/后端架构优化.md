# 后端架构与工程化优化文档

本文档统一汇总 LearnSphere AI 后端在架构、稳定性、性能和工程化方面的主要优化点，替代历史上分散的“后端架构优化.md”和“后端架构优化总结.md”双文档维护方式。

## 一、优化目标

当前后端优化主要围绕 4 个方向：

1. 可观测性
2. 缓存与响应性能
3. 异步处理能力
4. 稳定性保护与降级能力

## 二、可观测性

### 1. Actuator 与监控端点

已引入：

- `spring-boot-starter-actuator`

当前重点端点：

- `/actuator/health`
- `/actuator/metrics`
- `/actuator/prometheus`

作用：

- 便于直接观察数据库、Redis、磁盘与 JVM 健康状态

## 三、缓存与性能优化

### 1. 多级缓存策略

当前思路是把“数据库 / Redis / 本地缓存 / AI 生成”串成分层链路：

1. 先查本地高频缓存或 Redis
2. 再查数据库
3. 最后才走 AI 生成或重计算逻辑

### 2. Caffeine 二级缓存

已引入：

- `Caffeine`

适用场景：

- 系统字典
- 基础配置
- 高频读取但短时间不频繁变更的数据

收益：

- 减少 Redis 网络 IO
- 压低高频查询延迟

### 3. 缓存收益

典型收益包括：

- 相同请求从 AI 调用变为缓存读取
- Dashboard 等高频页面延迟明显下降
- AI 调用次数下降，成本更可控

## 四、异步处理能力

### 1. 适用场景

异步化主要用于：

- 长文本生成
- 复杂题目生成
- 耗时统计聚合
- 不适合阻塞主线程的后台任务

### 2. 基本思路

推荐链路：

1. 用户请求先拿到任务标识
2. 后台线程池异步处理
3. 结果写入缓存或数据库
4. 前端轮询或订阅查询处理结果

### 3. 线程池配置

应为 AI 与重任务单独配置线程池，避免主业务线程被耗尽。

## 五、稳定性与降级

### 1. 熔断与限时控制

当前后端稳定性治理重点包括：

- Resilience4j 熔断器
- TimeLimiter 超时控制
- 失败率阈值保护
- 自动恢复探测

典型效果：

- AI 服务异常时快速失败
- 避免线程池被长时间请求拖死
- 服务恢复后可自动探测并恢复正常调用

### 2. 降级策略

常见降级方向：

- 使用本地题库或数据库中已有数据
- 返回缓存数据
- 返回更轻量的占位结果或延迟执行结果

### 3. 异常处理精细化

全局异常处理已逐步区分：

- 业务异常
- 鉴权异常
- 配额与限流异常
- AI 调用异常

这样前端能获得更准确的状态码与错误语义。

## 七、当前收益总结

当前后端架构优化已经带来的直接收益包括：

- 系统健康状态更透明
- 高并发场景下缓存利用率更高
- AI 请求链路更容易做超时控制和降级
- 部署配置与生产敏感信息管理更规范
- 后续监控、告警和成本治理更容易落地

## 八、后续建议

如果继续推进后端架构优化，优先级更高的方向是：

1. 补数据库迁移登记机制
2. 继续补 Alertmanager / Grafana 实际落地配置
3. 为 AI、缓存、异步任务增加更清晰的维护说明
4. 继续清理历史文档中的旧部署方案和过时配置引用
