# 宝塔版本目录与发布记录模板

本文档用于约定 Linux 宝塔部署时的版本目录结构、命名方式和发布记录格式，目标是降低长期维护和回滚成本。

## 一、推荐目录约定

如果服务器允许调整目录结构，建议采用下面的方式：

```text
/www/wwwroot/learnsphere-backend/
├── current/                       # 当前运行版本目录
│   ├── learnsphere-ai-backend.jar
│   ├── requirements-voice.txt
│   └── config/
│       └── application-secret.properties
├── releases/                      # 历史版本目录
│   ├── 20260313-01/
│   │   ├── learnsphere-ai-backend.jar
│   │   ├── requirements-voice.txt
│   │   └── RELEASE.md
│   ├── 20260313-02/
│   └── ...
├── backups/                       # 配置和旧 Jar 备份
└── logs/                          # 部署或运行日志（可选）
```

## 二、版本命名建议

推荐使用以下格式：

```text
YYYYMMDD-序号
```

例如：

- `20260313-01`
- `20260313-02`
- `20260314-01`

这样做的好处：

- 一眼能看出发布时间顺序
- 不依赖 Git 分支名或临时口头描述
- 宝塔切换 Jar 路径时更容易定位版本

## 三、最小发布动作建议

每次发布时，建议按这个顺序：

1. 本地完成联调和构建
2. 在 `releases/新版本目录/` 上传本次 Jar 与附属文件
3. 记录本次发布说明
4. 将宝塔 Java 项目切到新版本 Jar
5. 验收通过后再决定是否同步更新 `current/`

如果当前还没有条件切到版本目录模式，也建议至少：

- 把当前运行 Jar 复制一份到 `backups/`
- 把上一个稳定 Jar 单独保留
- 把当前配置文件做一份备份

## 四、推荐发布记录模板

每次发布建议在版本目录下保留一个 `RELEASE.md`，可直接使用下面模板：

```md
# 发布记录

- 发布时间：2026-03-13 21:30
- 发布目录：20260313-02
- 发布人：
- 关联说明：

## 本次内容

- 是否包含用户前端更新：是 / 否
- 是否包含管理后台更新：是 / 否
- 是否包含后端更新：是 / 否
- 是否包含配置变更：是 / 否
- 是否包含数据库变更：是 / 否

## 产物

- Jar：learnsphere-ai-backend.jar
- 语音依赖：requirements-voice.txt（如有）
- 外置配置：application-secret.properties（是否更新）

## 验收结果

- 用户端首页：通过 / 未测 / 失败
- 管理后台：通过 / 未测 / 失败
- 登录接口：通过 / 未测 / 失败
- AI 接口：通过 / 未测 / 失败
- Redis 相关功能：通过 / 未测 / 失败

## 回滚基线

- 上一个稳定版本：20260313-01
- 回滚所需配置是否已备份：是 / 否
- 是否保留观察窗口：是 / 否

## 备注

-
```

## 五、推荐回滚记录模板

如果发生回滚，建议在同目录或运维记录中补一条：

```md
# 回滚记录

- 回滚时间：2026-03-13 22:10
- 回滚前版本：20260313-02
- 回滚后版本：20260313-01
- 回滚原因：
- 影响范围：
- 是否恢复配置文件：是 / 否
- 回滚后验证结果：
```

## 六、建议保留策略

至少建议保留：

- 最近 2 个稳定版本目录
- 最近 1 份可用配置备份
- 最近 1 次回滚记录

如果磁盘空间允许，可以保留：

- 最近 5 次发布目录
- 每次发布的 `RELEASE.md`
- 关键故障回滚记录

## 八、说明

本文档主要解决“版本怎么放、记录怎么写、回滚时怎么看”三个问题，不替代具体的构建和部署步骤。
