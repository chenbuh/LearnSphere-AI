# 发布产物与回滚说明

本文档面向当前项目的非 Docker 发布方式，重点说明：

- 本地打包后到底要上传什么
- 服务器上建议保留什么
- 宝塔环境下如何做最小化回滚

## 一、默认发布产物

如果当前采用“前后端合包”方式，核心发布产物是：

- `backend/target/*.jar`

如需本地语音能力，还需要同时准备：

- `requirements-voice.txt`

如采用外置配置，还需要准备：

- `config/application-secret.properties`

## 二、本地打包后建议确认的内容

在上传到宝塔之前，建议至少确认：

- Jar 已由最新代码打包生成
- 本次是否包含管理后台更新
- 本次是否包含前端静态资源更新
- 本次是否涉及配置字段变更
- 本次是否需要更新服务器上的外置配置文件

## 三、服务器目录建议

以宝塔部署目录为例，建议至少区分：

```text
/www/wwwroot/learnsphere-backend/
├── current/                 # 当前运行版本目录
│   ├── learnsphere-ai-backend.jar
│   ├── requirements-voice.txt
│   └── config/
│       └── application-secret.properties
├── releases/                # 历史发布产物
│   ├── 20260313-01/
│   └── 20260313-02/
└── logs/                    # 运行日志（如按目录保留）
```

如果暂时不想调整目录结构，至少也建议：

- 保留当前正在运行的 Jar 副本
- 保留上一个稳定版本 Jar
- 保留当前生效配置文件的备份

## 四、上传建议

### 方式 A：覆盖当前版本

适合低频更新场景：

1. 上传新的 Jar 到当前部署目录
2. 如有需要，更新 `requirements-voice.txt`
3. 如有需要，更新外置配置文件
4. 在宝塔 Java 项目中重启服务

### 方式 B：保留版本目录

适合更稳妥的发布方式：

1. 在 `releases/时间戳/` 下上传本次 Jar 和附属文件
2. 确认配置目录已就绪
3. 在宝塔 Java 项目中切换 Jar 路径到新版本
4. 重启并验收
5. 验收通过后再决定是否清理旧版本

## 五、建议保留的最小回滚材料

每次发布至少保留：

- 上一个稳定 Jar
- 上一个稳定配置文件副本
- 本次发布使用的配置说明
- 本次发布时间点和版本说明

如果本次涉及配置字段新增，建议额外保留：

- 新旧配置差异记录
- 本次发布前后的宝塔启动参数截图或文本记录

## 六、最小回滚步骤

如果新版本上线后出现问题，优先使用最小回滚：

1. 停止当前宝塔 Java 项目
2. 切回上一个稳定 Jar
3. 如本次改过配置，同时恢复上一个稳定配置文件
4. 启动旧版本
5. 重新检查首页、登录、关键接口、管理后台和 AI 接口

## 七、回滚触发条件建议

出现以下情况时，建议优先回滚而不是继续在线排障：

- 服务启动失败
- 登录链路异常
- 首页或管理后台白屏
- 核心接口大量 5xx
- 数据库或 Redis 连接异常
- AI 接口大面积失败且短时间无法修复

## 八、上线后保留建议

每次发布后，建议记录：

- 发布时间
- 使用的 Jar 名称或版本目录
- 是否包含管理后台更新
- 是否更新了外置配置
- 验收结果
- 是否需要保留回滚观察窗口
